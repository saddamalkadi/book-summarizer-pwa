تواصل معي بالعربية في جميع الشروحات والتقارير والملخصات النهائية.
أبقِ أسماء الملفات، الدوال، الحزم، الأوامر، ورسائل الخطأ بالإنجليزية كما هي.

مهم جدًا:
- لا تعيد بناء المشروع من الصفر.
- لا تحذف الميزات الحالية إلا إذا ثبت أنها مكررة أو ضارة أو معطلة، وعندها وثّق السبب قبل الحذف.
- لا تنفذ Refactor معماري واسع إلا إذا كان ضروريًا ومبررًا.
- فضّل التعديلات الآمنة التدريجية.
- الهدف هو تحويل المشروع الحالي إلى منصة جاهزة للنشر التجاري باحترافية عالية على الويب وAndroid وiPhone.
- اعمل كخبير Product Design + UI/UX + Frontend/Backend + Accessibility + Performance + Mobile Packaging + Commercial Readiness.

السياق الحالي:
المشروع وصل إلى المرحلة الرابعة، والمتبقي الآن هو المرحلة الخامسة: release hardening + polish + packaging + launch readiness.
أريدك أن تتعامل مع المرحلة الخامسة كمرحلة نهائية شاملة للإخراج التجاري، لا كتحسينات سطحية فقط.

الهدف النهائي:
تحويل التطبيق إلى منصة احترافية جاهزة للنشر تعكس قيمة وقدرات المنصة بوضوح، وتلبي متطلبات المستخدم النهائي، وتلتزم بالممارسات الحديثة في:
1) Product UX
2) Visual polish
3) Navigation clarity
4) Accessibility
5) Authentication usability
6) Voice chat reliability
7) Responsive quality
8) Performance
9) Release packaging for Web / Android / iPhone

معايير التصميم الإلزامية:
1. اعتمد progressive disclosure:
   - لا تعرض كل الخيارات المتقدمة في الشاشة الأولى.
   - اجعل البداية بسيطة وواضحة للمستخدم العادي.
   - انقل الإعدادات المتقدمة والحساسة إلى Advanced / Admin / Developer Settings.
2. على الهاتف، اجعل التنقل الرئيسي محدودًا وواضحًا:
   - لا تجعل الشريط الرئيسي مزدحمًا.
   - استخدم 3 إلى 5 وجهات عليا فقط كحد مستهدف.
   - انقل باقي الأقسام إلى More / Drawer / Secondary navigation.
3. اجعل أول شاشة تركز فقط على أهم ما يحتاجه المستخدم:
   - ابدأ محادثة
   - ارفع ملفات
   - شغّل مهمة جاهزة
   - اعرض باختصار القيمة الأساسية للمنصة
4. أصلح جميع مشاكل الترميز والنصوص والعرض فورًا، خصوصًا أي نص مكسور أو غير مقروء.
5. حسّن RTL والعربية والمحاذاة والمسافات والتسلسل البصري.
6. حسّن أحجام الأزرار والأهداف التفاعلية للمس بما يتوافق مع المتطلبات الحديثة.
7. اجعل تدفق تسجيل الدخول واضحًا وسهلًا وقابلًا للوصول.
8. اجعل الدردشة الصوتية موثوقة وواضحة الحالات.
9. حسّن الأداء وقابلية القياس قبل النشر.

المطلوب في هذه المرحلة:
A) Final Product Audit
- افحص الواجهة الحالية كاملة بعد أعمال المرحلة الرابعة.
- حدّد ما الذي ما زال يعيق الاحترافية التجارية.
- أنشئ ملف:
  - docs/05-final-product-audit.md
يتضمن:
- أهم مشاكل UX/UI المتبقية
- مشاكل navigation
- مشاكل المحتوى والترتيب
- مشاكل الوصولية
- مشاكل الأداء
- مشاكل login أو voice إن بقي منها شيء
- ما الذي يجب إنهاؤه قبل اعتباره launch-ready

B) Navigation & Information Architecture Polish
أعد تنظيم المنتج بحيث يصبح أوضح وأكثر تجارية:
- اجعل التنقل الرئيسي بسيطًا ومركزًا.
- اقترح ثم نفّذ بنية تنقل حديثة مناسبة للويب والموبايل.
- اجعل الوجهات العليا الأساسية فقط:
  1) Assistant / Chat
  2) Files
  3) Projects
  4) Tools أو Workflows
  5) More
- انقل المعرفة KB، Model Hub، الإعدادات المتقدمة، البوابة، مفاتيح API، إعدادات OCR/Cloud، وخيارات المطور إلى مستوى ثانوي.
- إذا كانت بعض الأقسام أنسب كصفحات داخلية أو accordions أو drawers أو tabs ثانوية، طبّق ذلك.
- امنع تكدس الأوامر والإعدادات في واجهة واحدة.

C) Home / Landing Workspace Redesign
أعد تصميم الشاشة الأولى داخل التطبيق لتصبح:
- أوضح
- أبسط
- أعلى قيمة تجارية
- أفضل للمستخدم الجديد

اجعل أول شاشة تحتوي على:
- عنوان قيمة واضح جدًا للمنصة
- وصف قصير يشرح ما تفعله المنصة
- 3 إجراءات رئيسية فقط:
  - ابدأ محادثة
  - ارفع ملفات
  - شغّل مهمة جاهزة
- بطاقات استخدامات عالية القيمة مثل:
  - تحليل ملف
  - بحث عميق
  - استخراج نص من PDF
  - إنتاج تقرير/مخرج احترافي
- أزل أو أخفِ أي تشويش بصري غير ضروري من أعلى الصفحة

D) Visual Design Upgrade
أريد المظهر النهائي أن يعكس منصة قوية وعصرية وتجارية.
قم بما يلي:
- توحيد الألوان والظلال والحواف والمسافات
- تحسين hierarchy البصري
- توحيد أنماط الأزرار والبطاقات والحقول
- تحسين typography العربية والإنجليزية
- إزالة أي مظهر تجريبي أو مزدحم
- تحسين empty states
- تحسين loading states
- تحسين error states
- تحسين success feedback
- تحسين microcopy داخل الواجهة
- اجعل المنتج يبدو premium, modern, trustworthy

E) Login Hardening & UX Validation
حتى لو تم إصلاح login سابقًا، أعد تدقيقه كمرحلة إطلاق:
- اختبر تدفق الدخول كاملًا
- اختبر حالات الفشل
- اختبر إعادة المحاولة
- اختبر وضوح رسائل الخطأ
- اختبر الجلسة logout / session expiry / refresh behavior
- حسّن accessibility في المصادقة
- لا تعتمد على اختبارات إدراكية غير ضرورية
- اجعل استعادة الحساب أو إعادة تسجيل الدخول مفهومة وواضحة

أنشئ:
- docs/06-auth-release-readiness.md

F) Voice Chat Hardening
حتى لو تم إصلاح voice chat سابقًا، أعد تدقيقه كمرحلة إطلاق:
- اختبر permission flow
- اختبر start / stop / processing / error / retry states
- اختبر fallback إلى النص
- اختبر mobile usability
- اختبر ما إذا كان المستخدم يفهم حالة النظام في كل لحظة
- أصلح أي UI ambiguity أو latency feedback issue
- وثّق root causes المتبقية إن وجدت

أنشئ:
- docs/07-voice-release-readiness.md

G) Accessibility Pass (Required)
نفّذ مراجعة وصولية حقيقية على الواجهة الأساسية:
- أحجام الأزرار والأهداف التفاعلية
- تباين النصوص
- keyboard/focus states
- labels للأيقونات والأزرار
- form fields and errors
- auth accessibility
- dialog and modal behavior
- screen-reader hints where appropriate
- لا تكتفِ بادعاء التوافق، بل طبّق التحسينات فعليًا واذكرها

أنشئ:
- docs/08-accessibility-pass.md

H) Performance & Web Readiness
أريد تدقيق أداء حقيقي قبل النشر:
- راجع bundle / asset loading / render blocking / layout shifts
- حسّن loading behavior
- خفف أي JS/CSS غير ضروري
- أصلح أي مشاكل تسبب بطءًا ملحوظًا
- أضف أو فعّل قياس Web Vitals إن كان غير موجود
- وثّق كيف يمكن متابعة LCP / INP / CLS
- تأكد من أن النسخة الويب قابلة للقياس والتحسين بعد النشر

أنشئ:
- docs/09-performance-and-web-vitals.md

I) Commercial Readiness Pass
أريد أن يبدو المنتج جاهزًا تجاريًا:
- راجع onboarding
- راجع التسلسل المنطقي لاكتشاف المزايا
- راجع اللغة المستخدمة داخل المنتج
- اجعل القيمة الأساسية واضحة من أول دقيقة
- تأكد أن المستخدم الجديد لا يواجه تعقيدًا غير ضروري
- راجع هل المنتج يبيع نفسه بصريًا ووظيفيًا
- حسّن copywriting داخل التطبيق عند الحاجة
- لا تجعل المنتج يبدو كأداة داخلية للمطورين

أنشئ:
- docs/10-commercial-readiness.md

J) Release Packaging
في النهاية جهّز ثلاث نسخ:
1) Web-ready
2) Android build-ready
3) iPhone build-ready

المطلوب هنا:
- تأكد أن نسخة الويب تعمل وتصلح للنشر
- تأكد أن Android wrapper / build path جاهز
- تأكد أن iPhone wrapper / build path جاهز
- إن وجدت خطوة خارجية خاصة بـ Apple/Xcode/Certificates/Provisioning، وثّقها بوضوح ولا تخفها
- أنشئ قائمة build & release واضحة جدًا

أنشئ:
- docs/11-release-build-guide.md
- docs/12-final-launch-checklist.md

معايير القبول النهائية:
لا تعتبر المرحلة الخامسة مكتملة إلا إذا تحقق التالي:
1) الواجهة أصبحت أبسط وأوضح وأكثر تجارية
2) الشاشة الأولى تبرز القيمة ولا تشتت المستخدم
3) التنقل الرئيسي لم يعد مزدحمًا
4) الإعدادات المتقدمة أصبحت في مكان ثانوي مناسب
5) لا توجد نصوص مكسورة أو مشاكل ترميز
6) login محسّن ومفهوم وقابل للوصول
7) voice chat محسّن وواضح الحالات وموثوق
8) responsive behavior جيد على الجوال والويب
9) accessibility تحسنت بشكل ملموس وموثق
10) الأداء وWeb Vitals موثقة
11) توجد ملفات توثيق نهائية واضحة للبناء والإطلاق
12) المشروع صار launch-ready بوضوح

طريقة التنفيذ:
- ابدأ بالتدقيق النهائي ثم أصلح أهم المشاكل أولًا.
- لا تقفز مباشرة إلى التغليف قبل صقل المنتج.
- بعد كل مجموعة تعديلات، أعطني ملخصًا بالعربية:
  - ما الذي وجدته
  - ما الذي أصلحته
  - ما الذي بقي
- إذا احتجت secrets أو إعدادات خارجية، اطلبها بوضوح.
- إذا احتجت تعديلًا كبيرًا في بنية التنقل، نفّذه تدريجيًا مع الحفاظ على الميزات الحالية.

ابدأ الآن بتنفيذ المرحلة الخامسة كاملة وفق هذا الترتيب:
1) final audit
2) navigation & home redesign
3) login and voice hardening
4) accessibility pass
5) performance pass
6) release packaging
ثم أعطني تقريرًا مرحليًا بالعربية بعد الانتهاء من كل جزء رئيسي.
