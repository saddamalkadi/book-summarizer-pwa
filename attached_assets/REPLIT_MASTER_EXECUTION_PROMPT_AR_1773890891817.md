# تعليمات تنفيذ رئيسية إلى Replit Agent

أنت تعمل الآن كخبير تقني ومنتجي وتجاري على هذا المستودع. مهمتك ليست إعادة بناء المشروع من الصفر، بل **تحويله إلى منصة جاهزة للنشر تجاريًا** عبر الويب وAndroid وiPhone، مع الحفاظ على جوهر المشروع الحالي وتعزيز خدماته الحالية.

## القاعدة الأساسية
- لا تعيد بناء المشروع من الصفر.
- لا تحذف الخدمات الموجودة إلا إذا كانت معطلة أو مكررة ويجب استبدالها بحل أفضل موثق.
- اعمل على مراحل واضحة، مع تقارير قصيرة بعد كل مرحلة.
- قبل أي refactor كبير: شخّص السبب الجذري للمشاكل الحالية أولًا.
- أي تغيير كبير يجب أن يكون قابلاً للاختبار والتراجع.
- حافظ على دعم العربية وRTL بشكل احترافي كامل.
- حافظ على التوافق مع PWA وCapacitor إن كان موجودًا.

## الهدف النهائي
إنتاج منصة احترافية حديثة جاهزة تجاريًا، تحقق ما يلي:
1. إصلاح جميع المشاكل الحرجة الحالية، خصوصًا:
   - تسجيل الدخول
   - الدردشة الصوتية
   - الأعطال أو الأخطاء الظاهرة في الواجهة
   - مشاكل التبويبات والقوائم والتنقل
2. إعادة تنظيم الواجهة والتجربة لتصبح بمستوى تجاري حديث.
3. تعزيز الخدمات الموجودة حاليًا بدل إضعافها.
4. تجهيز 3 نواتج نهائية:
   - نسخة ويب جاهزة للنشر
   - نسخة Android جاهزة للبناء/النشر
   - نسخة iPhone جاهزة للبناء/النشر
5. توثيق ما تم تغييره وكيفية الاختبار والنشر.

## الأولويات المطلقة
رتّب العمل بهذا التسلسل الصارم:

### المرحلة 1: التدقيق والتشخيص
نفّذ أولًا فقط:
- تحليل بنية المشروع كاملة
- تحديد نقاط الدخول الأساسية
- تحديد آلية تشغيل الويب داخل Replit
- تشخيص سبب فشل تسجيل الدخول بدقة
- تشخيص سبب فشل الدردشة الصوتية بدقة
- حصر أخطاء الواجهة، الترميز، التنقل، والأداء
- حصر الخدمات الحالية الموجودة في المشروع وتقييم حالتها

#### مخرجات المرحلة 1
أنشئ الملفات التالية داخل `docs/`:
- `docs/01-baseline-audit.md`
- `docs/02-bug-diagnosis-login-voice.md`
- `docs/03-feature-inventory.md`
- `docs/04-ui-ux-issues.md`

ثم توقّف وقدّم تقريرًا موجزًا قبل بدء refactor كبير.

### المرحلة 2: إصلاح المشاكل الحرجة
بعد التشخيص، أصلح أولًا فقط المشاكل الحرجة التالية:

#### أ. تسجيل الدخول
يجب:
- تحديد root cause بدقة
- إصلاح تدفق login/logout/session
- إصلاح token handling / storage / refresh إن وجد
- منع حالات التعليق أو الفشل الصامت
- إضافة رسائل خطأ واضحة ومهنية للمستخدم
- ضمان عمل تسجيل الدخول على الويب أولًا، ثم توافقه مع Android وiPhone

#### ب. الدردشة الصوتية
يجب:
- تشخيص تدفق الميكروفون والأذونات والتسجيل والإرسال والاستقبال
- إصلاح مشاكل audio capture / playback / permissions / state
- إصلاح تعطل الواجهة أثناء الصوت
- ضمان fallback واضح عند فشل ميزة معينة
- تحسين تجربة الاستخدام في mobile وweb

#### ج. الاستقرار العام
يجب أيضًا:
- إصلاح أي أخطاء console ظاهرة
- إصلاح مشاكل الترميز أو النصوص التالفة
- إصلاح المشاكل التي تمنع التشغيل السليم

#### مخرجات المرحلة 2
- تحديث `docs/05-critical-fixes.md`
- قائمة قبل/بعد لكل bug تم إصلاحه
- خطوات اختبار واضحة

---

### المرحلة 3: إعادة الهيكلة بدون كسر السلوك
بعد استقرار النسخة:
- لا تغيّر السلوك الأساسي إلا عند الحاجة
- قسّم الملفات الكبيرة إلى modules منطقية
- افصل طبقات المشروع إلى:
  - UI
  - state management
  - auth
  - chat
  - voice
  - providers/services
  - files/knowledge/tools
- قلّل التشابك بين الواجهة والمنطق الخلفي
- نظّف الأكواد المكررة
- حسّن التسمية والتنظيم

#### المطلوب هنا
- الحفاظ على التوافق الحالي قدر الإمكان
- عدم إدخال framework جديد إلا إذا كان ذلك ضروريًا جدًا ومبررًا
- إذا قررت عدم استخدام framework جديد، حسّن البنية الحالية باحتراف

#### مخرجات المرحلة 3
- `docs/06-architecture-refactor.md`
- مخطط بنية الملفات الجديدة
- شرح الملفات المنقولة أو المقسّمة

### المرحلة 4: تحسين التصميم والمنتج التجاري
أعد تصميم التجربة لتكون **بمستوى SaaS/AI Workspace حديث**.

#### المطلوب تصميميًا
- واجهة نظيفة، حديثة، احترافية
- وضوح بصري عالي
- تنظيم أفضل للتبويبات والقوائم
- تقليل الازدحام في الصفحة الأولى
- جعل أول تجربة للمستخدم واضحة وسهلة
- إبراز الوظائف الأساسية أولًا
- نقل الخيارات المتقدمة إلى مواضع ثانوية منطقية
- تحسين الهوية البصرية والأزرار والبطاقات والحالات الفارغة
- تحسين responsive behavior
- ضمان RTL عربي ممتاز

#### المطلوب وظيفيًا في UX
- مسار واضح لأول استخدام
- Onboarding مبسط
- تسميات واضحة للخدمات
- تنقل منطقي بين:
  - الدردشة
  - الملفات
  - المعرفة / KB
  - الأدوات / الوكلاء
  - الإعدادات
- تقليل الإرباك

#### التوجه التجاري المطلوب
اجعل المنصة تبدو كمنتج تجاري جاهز للبيع والاستخدام، وليس أداة تجريبية.

#### مخرجات المرحلة 4
- `docs/07-ui-ux-redesign.md`
- قائمة التحسينات المرئية والتنقلية
- لقطات أو وصف واضح للحالات الرئيسية

---

### المرحلة 5: تعزيز الخدمات الحالية
لا تحذف الخدمات الحالية. المطلوب هو **تعزيزها** وتحسين موثوقيتها وتجربتها.

#### افحص وادعم ما هو موجود مثل:
- المحادثة الذكية
- الملفات والمرفقات
- البحث/المعرفة/KB/RAG
- الدردشة الصوتية
- الأدوات أو الوكلاء
- التحويلات أو OCR أو الخدمات السحابية إن كانت موجودة
- إدارة المشاريع أو السجلات أو المحفوظات إن كانت موجودة

#### المطلوب
- توحيد عرض الخدمات
- تحسين تسمياتها
- ضمان عدم تضاربها
- إضافة fallbacks حيث يلزم
- تحسين الأداء والرسائل وحالات الفشل
- تعزيز أي خدمة ناقصة لكن موجودة أصلًا

#### مخرجات المرحلة 5
- `docs/08-service-enhancement-report.md`

### المرحلة 6: الجاهزية للنشر
اجعل المشروع جاهزًا للنشر الاحترافي.

#### نسخة الويب
- تأكد من build/run واضح
- أصلح كل ما يمنع النشر
- أضف تعليمات نشر واضحة
- تأكد من عمل الـ PWA إن كان مدعومًا
- راجع الأيقونات والمانيفست والخدمات المرتبطة

#### نسخة Android
- افحص تكامل Capacitor/Android إن وجد
- أصلح ملفات التهيئة اللازمة
- تأكد أن المشروع build-ready
- وثّق خطوات الإخراج النهائي
- أنشئ ملف توثيق خاص بالأندرويد

#### نسخة iPhone
- افحص تكامل Capacitor/iOS إن وجد
- جهّز المشروع ليكون build-ready
- وثّق بوضوح أي خطوات تتطلب macOS/Xcode/Apple certificates
- لا تدّعِ اكتمال توقيع iOS إذا كانت هناك خطوة خارجية لازمة
- سلّم نسخة جاهزة تقنيًا للمرحلة الأخيرة

- `docs/11-ios-release-readiness.md`

## قيود مهمة جدًا
- لا تُجرِ تغييرات تدميرية دون توثيق.
- لا تحذف ميزة قائمة فقط لأنها صعبة الصيانة.
- لا تكتب شيفرة كثيرة غير ضرورية إذا كان الإصلاح الأبسط أفضل.
- لا تغيّر أسماء ومسارات حيوية دون سبب واضح.
- لا تترك أخطاء lint/build/console واضحة دون معالجة أو توثيق.
- لا تكتفِ بالحل النظري؛ اختبر عمليًا داخل بيئة Replit قدر الإمكان.

## معايير القبول النهائية
لا تعتبر المهمة مكتملة إلا إذا تحقق الآتي:

### 1) الاستقرار
- تشغيل المشروع بنجاح داخل Replit
- عدم وجود أخطاء حرجة تمنع الاستخدام
- تسجيل الدخول يعمل
- الدردشة الصوتية تعمل أو يوجد fallback موثق وواضح

### 2) جودة المنتج
- الواجهة حديثة ومنظمة
- التبويبات والقوائم منطقية
- الصفحة الرئيسية غير مزدحمة
- الاستخدام أوضح وأسهل
- العربية وRTL سليمان

### 3) جودة الهندسة
- بنية الكود أوضح من السابق
- تقليل التعقيد والملفات المتضخمة
- فصل معقول للمسؤوليات
- الحفاظ على السلوك الحالي أو تحسينه دون كسر

### 4) الجاهزية التجارية
- نسخة ويب جاهزة
- Android build-ready
- iPhone build-ready مع توثيق أي خطوة خارجية لازمة
- توثيق نهائي واضح

## نقطة حاسمة
ابدأ الآن بالمرحلة الأولى فقط:
- شغّل المشروع داخل Replit
- افحص root causes
- أنشئ ملفات التدقيق المطلوبة
- أصلح فقط ما يلزم لفهم المشكلة إن كان التشغيل متعذرًا

بعد ذلك، انتقل إلى إصلاح login وvoice chat أولًا قبل أي تحسين تجميلي كبير.
